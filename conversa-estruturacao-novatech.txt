[ASSISTENTE] — GOVERNANCA DE SPECS (COMPLETA)
--------------------------------------------------------------------------------

PRINCIPIO CENTRAL: Specs nao sao documentos — sao contratos. Nenhuma linha de
codigo de producao existe sem uma spec aprovada que a originou.

OS TRES ARTEFATOS SDD:

requirements.md — O QUE
  Dono: Product Specialist
  Revisor: Tech Lead (valida viabilidade tecnica)
  Aprovador: TL + DM
  Conteudo: outcomes, scope boundaries, constraints, ADRs referenciadas,
             verification criteria testáveis. SEM decisoes de implementacao.

plan.md — O COMO
  Dono: Tech Lead
  Revisor: PS + Dev Senior
  Aprovador: Product Specialist
  Conteudo: abordagem tecnica, decisoes de arquitetura locais, dependencias
             externas, riscos. SEM tasks.

tasks.md — UNIDADES ATOMICAS
  Dono: Dev (+ GitHub Copilot)
  Revisor: Tech Lead
  Aprovador: Tech Lead
  Conteudo: ID unico, descricao, criterio de aceite, dependencias, estimativa (P/M/G),
             responsavel. Task atomica = concluivel em maximo 1 dia.

CONVENCOES DE NOMENCLATURA:

Cabecalho obrigatorio:
  <!-- spec: query-endpoint/requirements | version: 1.2 | status: aprovada
       autor: [PS] | aprovado-por: [TL] | aprovado-em: 2026-06-11 -->

Versionamento semantico simplificado:
  Major (1.x→2.x): mudanca de escopo → reabre Gate 1
  Minor (1.1→1.2): clarificacao → nao reabre gate

IDs de tasks:
  INGEST-T001 | QUERY-T001 | FEEDBACK-T001 | BOT-T001 | PAINEL-T001

Titulo de PR:
  [SPEC][query-endpoint/requirements][minor] Clarifica VC-03
  [CODE][query-endpoint][QUERY-T003] Implementa validacao de input via Zod

CICLO DE VIDA:
  Rascunho → Em Revisao → Aprovada → Em Implementacao → Validada → Obsoleta

RASTREABILIDADE:
  - Toda mudanca passa por PR (nao editar main diretamente)
  - Secao Changelog obrigatoria em toda spec
  - PR de codigo referencia ID da task no titulo

CHANGE MANAGEMENT:
  Minor: PR com label spec-minor, revisao em 4h, sem reabrir gate
  Major (requirements): TL + DM, Gate 1 reaberto, tasks afetadas = "blocked"
  Major (plan): PS + Dev Senior, novo tasks.md obrigatorio antes de retomar
  Template de Change Request: motivacao, o que muda, o que NAO muda,
                               tasks afetadas, impacto em prazo, aprovacoes

REGRAS PARA AGENTES DE IA:
  DEVE: referenciar ID da task no codigo (// implements QUERY-T003)
  DEVE: usar template do modulo em /specs/{modulo}/
  NAO DEVE: criar specs fora da estrutura definida
  NAO DEVE: modificar status no cabecalho da spec
  NAO DEVE: iniciar geracao de tasks sem requirements aprovado
