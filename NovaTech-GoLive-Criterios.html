CAMADA 1 DE 5

Tool Orchestration

Coordenação entre Azure AI Search, Azure OpenAI e o Bot Framework — garantir que as ferramentas conversam entre si de forma confiável.

4Bloqueantes
2Desejáveis
Bloqueante
Pipeline de ingestão processa os 847 documentos válidos sem erro
Sem base indexada corretamente, não há o que recuperar.
Bloqueante
Query endpoint orquestra busca → geração → resposta em sequência confiável, sem timeout silencioso
Falha na orquestração quebra a experiência completa do atendente.
Bloqueante
Bot do Teams consome o query endpoint sem perda de payload
Já validado em staging com 5 atendentes-piloto — manter monitorado até o go-live.
Bloqueante
Fallback definido quando o Azure OpenAI não responde dentro do SLA
Sem isso, o atendente fica esperando indefinidamente sem nenhum feedback.
Desejável
Re-tentativa automática em caso de falha transitória da API
Mitiga instabilidade pontual, mas não impede a operação manual no go-live.
Desejável
Roteamento dinâmico entre múltiplos índices do Azure AI Search
Não existe necessidade hoje — apenas 1 índice está ativo no projeto.
CAMADA 2 DE 5

Verification Loops

Verificação automática de outputs — a camada mais crítica diante dos 12% de respostas incorretas identificadas em teste.

6Bloqueantes
1Desejável
Bloqueante
Structured outputs implementado: resposta sempre em JSON com campos obrigatórios (answer, source_document, confidence_score)
Hoje a resposta é texto livre — é a causa raiz do problema de fonte ausente.
Bloqueante
Respostas sem source_document preenchido são rejeitadas programaticamente antes de chegar ao atendente
Sem isso, alucinação sem fonte passa direto, como já ocorreu nos 12% de erro identificados.
Bloqueante
Taxa de alucinação validada em golden dataset abaixo de 5% antes do go-live
12% atual é inaceitável para a demo de diretoria — meta mínima de redução.
Bloqueante
Verificação automática de que o source_document citado existe na lista de 847 documentos válidos
Evita citação de documento descartado ou inexistente — alucinação de fonte.
Bloqueante
Verificação cruzada de versão quando dois documentos do mesmo tópico estão indexados (ex: PROC-042 v1 vs v2)
Risco documentado desde a fase de discovery — sem isso, mistura de versões é praticamente garantida.
Bloqueante
Verificação de idioma da resposta — sempre português, independente do idioma da pergunta
Identificado como falha real em teste: resposta em inglês para pergunta em inglês.
Desejável
Comparação automática de respostas similares para detectar inconsistência entre sessões diferentes
Útil para monitoramento contínuo, não crítico para a demo.
CAMADA 3 DE 5

Context & Memory

Manutenção de contexto entre interações — verificar que a implementação é fiel à estratégia já decidida na ADR-0002.

3Bloqueantes
2Desejáveis
Bloqueante
Context budget implementado conforme ADR-0002 — 4K system + 8K chunks + 3 turnos de histórico
Decisão já tomada e documentada — verificar que a implementação é fiel à ADR.
Bloqueante
Janela deslizante de histórico ativa para evitar context rot em sessões longas
Sem isso, sessões com múltiplas perguntas degradam silenciosamente.
Bloqueante
Sistema não assume dados não informados pelo atendente (ex: região do frete)
Falha real identificada em teste — o modelo assumiu "Sudeste" sem o atendente informar.
Desejável
Reset de contexto disponível para o atendente iniciar nova consulta sem histórico anterior
Boa prática de UX, mas contornável manualmente abrindo nova conversa.
Desejável
Sumarização automática de histórico em sessões acima de 5 turnos
Mitigação adicional de context rot, não crítica com a janela deslizante já implementada.
CAMADA 4 DE 5

Guardrails

Limites que o sistema não pode ultrapassar — inclui os dois pontos obrigatórios de revisão humana (HITL) antes da resposta chegar ao atendente.

6Bloqueantes
1Desejável
Bloqueante
Guardrails de produto (DEVE / NÃO DEVE / QUANDO EM DÚVIDA) implementados no prompt e validados em teste
Já formalizados pelo Product Specialist na fase de estruturação — falta validar que estão em vigor no comportamento real.
Bloqueante
Human-in-the-Loop
Respostas de baixa confiança sobre carga perigosa são bloqueadas e escaladas para supervisor antes de chegar ao atendente
Tema com risco de segurança e de responsabilidade civil para a NovaTech.
Bloqueante
Human-in-the-Loop
Respostas de baixa confiança sobre devolução, frete especial ou SLA contratual são sinalizadas como "verificar com supervisor" em vez de apresentadas como definitivas
Reduz o risco de repasse de informação incorreta em temas com impacto financeiro direto ao cliente.
Bloqueante
Sistema nunca inventa tier de cliente inexistente — sempre redireciona para os 3 tiers documentados (Gold/Silver/Standard)
Cenário clássico de alucinação, já mapeado desde a fase de discovery.
Bloqueante
Sistema recusa responder fora do escopo de logística
Validado em teste com sucesso — manter como guardrail formal e monitorado.
Bloqueante
Mensagem padrão de "não encontrei informação" ativa e testada em todos os gaps documentais conhecidos
Base do comportamento esperado documentado desde o Anexo B — sem isso, risco de alucinação em gaps.
Desejável
Limite de taxa (rate limiting) por atendente para evitar abuso ou loop de perguntas automatizadas
Risco operacional baixo com 5 atendentes-piloto; relevante para escala futura.
CAMADA 5 DE 5

Observability

Visibilidade do que acontece em produção — a camada ainda não construída até o momento, e necessária para que a diretoria confie no sistema na demo.

4Bloqueantes
2Desejáveis
Bloqueante
Logging de toda interação (pergunta, resposta, fonte citada, confidence score, latência) sem dados sensíveis do atendente
Sem isso, é impossível investigar os 12% de erro já identificados ou futuros incidentes.
Bloqueante
Dashboard mínimo com taxa de erro, latência média e volume de perguntas/dia
Visibilidade mínima necessária para a diretoria confiar no sistema na demo.
Bloqueante
Alerta automático quando a taxa de respostas sem fonte citada excede o threshold definido
Conecta diretamente ao problema identificado nos testes internos — sem alerta, o problema só seria descoberto pelo cliente.
Bloqueante
Retenção de logs por período definido com tratamento adequado de LGPD para dados do atendente
Risco legal se não tratado — o atendente é colaborador da NovaTech e seus dados precisam de tratamento adequado.
Desejável
Rastreamento de quais documentos são mais consultados e quais perguntas não têm resposta
Valioso para melhoria contínua pós-go-live, não bloqueia a operação inicial.
Desejável
Painel de observabilidade acessível à NovaTech, não apenas à DB1
Aumenta transparência e confiança, mas pode ser entregue na primeira iteração pós-go-live.
